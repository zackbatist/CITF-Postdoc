What is qc-atelier

qc-atelier is a personal practitioner toolkit built on top of the qc CLI (qualitative-coding on PyPI). It is a single researcher’s tool, connected to a workshop on citational practices in Viking Age archaeology (“Citational Politics and Practices in Viking Age Archaeology and Beyond”, University of Oslo, in collaboration with Julie Lund and Isak Roalkvam, seeded from Lund & Sindbæk 2022 “Crossing the Maelstrom”).

The toolkit is the primary active technical focus of this researcher’s post-doctoral work. The code lives in CITF-Postdoc on GitHub (public). Data lives in qc/, private/, and data/ as git submodules on a university-managed GitLab instance (private).

The tools:

Tool Role Status
qc-scheme Passive — documents codes, manages snapshots and history Active
qc-refactor Active — structural operations on the code system Active
qc-viz Passive — visual corpus reader Active
qc-align Active — LLM-assisted code system audit and maintenance In development
qc-unfold Active — LLM-assisted corpus meaning-making Planned
qc-trace Passive — cross-tool audit timeline Designed, not started

The key distinction: qc-scheme and qc-viz are passive/responsive — they read and display. qc-refactor is active — it executes structural changes. qc-align and qc-unfold are active — they use the LLM to suggest and execute changes. This separation is intentional and must be maintained.

The analytical axis: qc-align operates on the code system, using the corpus as evidence. qc-unfold operates on the corpus text, using the code system as an analytical lens. They represent different phases: qc-align calibrates the instrument; qc-unfold uses it.

Technical environment: Python 3.14 on macOS ARM, .venv virtual environment, GROBID 0.8.1 via Docker, Ollama with qwen3:35b (thinking mode disabled via "think": false). qc installed globally via pipx at /Users/zackbatist/.local/bin/qc.

The qc CLI — upstream dependency

qc stores code systems as bare YAML trees and coding applications in a SQLite database.

Core data structures:

File Description
codebook.yaml Code system as bare YAML tree (names + hierarchy only)
qualitative_coding.sqlite3 SQLite database — live source of truth for all code applications
settings.yaml qc project configuration
qualitative_coding.log Structured operation log written by qc

qc-atelier exports (generated by pre-render script, not by qc itself):

File Description
qc/json/*.json Per-document coding exports: [{code, document, line, text}]
qc/corpus/*.txt Source interview transcripts

Critical: qc/json/*.json are render-time exports by qc-atelier, not qc’s live data. qualitative_coding.sqlite3 is qc’s live source of truth. qc-atelier must never write to the SQLite database directly, with one exception: qc-align code propagation (batch CodedLine inserts, always preceded by a SQLite backup and dryrun).

qc data model (verified from source — corpus.py, models.py):

The database uses SQLAlchemy over SQLite. Key tables: Code (name as primary key), Coder, Document, DocumentIndex, Location (line range), CodedLine (core coding record: code + coder + line).

qc codes rename reassigns CodedLine.code_id in the database then calls update_codebook() to sync codebook.yaml. This is also how merge works — multiple old codes renamed to one new code. codebook.yaml is therefore derived from the database, not the source of truth.

What qc does NOT have:

  • qc codes delete — no command exists to remove a code and its corpus applications
  • qc codes move — no command exists to reassign a code’s parent in the hierarchy
  • qc codes apply — no command exists to programmatically apply a code to a line

Tools

qc-scheme

Role: Passive. The primary documentation and review interface. Does not modify code structure.

qc-scheme enriches the bare YAML code system with what qc cannot store: scope definitions, rationale, usage notes, provenance, status, and canonical corpus examples, kept in codebook.json alongside an append-only edit history. It makes the evolution of the code system deliberate and auditable.

What it does:

  • GUI editor for per-code documentation: scope, rationale, usage_notes, provenance, status, pinned_examples
  • Per-code and document-level history (field edits + refactor events from qc-refactor, merged and sorted)
  • Snapshot system: named checkpoints and forks in qc/snapshots/
  • Cross-snapshot diffing (Fields and JSON views)
  • Multi-code comparison (table and cards views, bulk status changes)
  • Export: YAML, MD, QMD, HTML, CSV, PDF

UI reference:

Sidebar — full indented code tree, refreshed from server at runtime. Click to select; Shift/Cmd-click for multi-select. Pip icons for export selection. Status indicators (deprecated → dimmed, experimental → italic). Use counts per node.

Documentation tab — editable fields: Scope, Rationale, Usage notes, Provenance, Status. Autosaves on blur or 2.5s inactivity. Debounced 800ms server save. Parent field removed — structural moves belong to qc-refactor.

Examples tab — corpus excerpts alongside pinned canonical examples. Pin/unpin controls.

History tab — per-code field edit timeline. Click to inspect; click two to diff. Uses earliest _log entry’s from value as baseline (replaces removed _baseline).

Multi-selection toolbar — bulk status change, comparison table, cards view.

Topbar — mode switcher: Codebook / History / Snapshots.

Snapshot system:

Snapshots are named directories in qc/snapshots/ containing codebook.yaml and codebook.json at the moment of creation. lineage.json maps each snapshot to its parent and note. .working_parent tracks the currently loaded snapshot. Opening a snapshot fetches the snapshot’s codebook.yaml from the server and rebuilds the tree at runtime — no re-render needed.

No prior snapshot:
  snapshot → codebook_TIMESTAMP_label
Prior snapshot exists:
  snapshot → codebook_TIMESTAMP_label     (same chain)
  fork    → codebook-{segment}_TIMESTAMP_HASH4   (new chain)

History and diffing:

All per-code _log entries and all changelog events (including refactor events from qc-refactor) merged and sorted newest first. A/B colour coding: first clicked = teal, second = amber. Fields view: side-by-side before/after per changed field. JSON view: full serialised diff with accordion for long unchanged runs (>30 lines), no LCS size limit.

qc-refactor

Role: Active. The only tool that modifies code structure.

What it does:

  • Stage rename, merge, move, and deprecate operations in a unified queue
  • Preview the post-operation code tree and affected corpus segments before committing
  • Inspect the generated bash script (rename/merge only; move/deprecate are server-side)
  • Execute the queue with a required session note
  • Snapshot creation and loading via Snapshots mode in the topbar
  • Write provenance notes and changelog entries back to codebook.json

Operations:

Type Mechanism Corpus affected?
Rename qc codes rename A B subprocess Yes
Merge qc codes rename A B C subprocess Yes
Move Direct codebook.yaml edit (server-side) No
Deprecate status: "deprecated" in codebook.json No
Delete Not implemented — no qc codes delete exists —

Session note: Required before execution. Written to the refactor changelog entry as summary and seeds the provenance field of affected codes. Not used as snapshot label — snapshots are created manually.

Orphan cleanup: After successful rename/merge, server runs qc codes stats -zu 0 to identify zero-use codes, intersects with operation sources, and strips confirmed orphans from codebook.yaml via _do_remove. This is a workaround for upstream issue #94 (qc codebook --clean not yet implemented).

codebook.json sync: On rename, the code key is updated and children reparented. On merge, source keys are removed and children reparented to target. On move, the parent field is updated directly.

Queue: Unified — all op types in a single list with type badges. Operations can be edited after staging (edit button repopulates the form). Bulk move uses a multi-select checklist with collapsible tree branches.

Snapshots mode: Topbar toggle (Refactor | Snapshots). Full-panel snapshot UI with create zone (label, description, filename preview) and snapshot feed. Loading a snapshot copies codebook.yaml and codebook.json over working files and refreshes the runtime tree.

qc-viz

Role: Passive. Visual corpus reader with inline code tags and filtering. No server required at runtime.

What it does:

  • Renders each interview transcript line by line with inline coloured code tags
  • Filtering by category (Johnny Decimal prefix), individual code, speaker, coded/uncoded status
  • Whitelist/blacklist system in the Lua filter (e.g. O_* open-coding codes excluded by default)
  • Export: CSV, JSON, Excel, HTML, PDF — with filter parameters documented in the output
  • Filter state persisted to localStorage under qc.viz.state

Code schema priority:

  1. codebook.yaml (if codebook.enabled: true in config)
  2. code_schema.categories in qc-atelier-config.yaml
  3. Built-in default schema
  4. Auto-generated from first code in each prefix

codebook.enabled defaults to false — not yet validated against the actual codebook.yaml structure.

Speaker column width: Computed dynamically as max(80, min(200, max_speaker_length × 8)) px.

CDN dependency: SheetJS and jsPDF loaded unconditionally. CSV/JSON/HTML export work offline; Excel and PDF require network.

qc-align

Role: Active. LLM-assisted audit and maintenance of the code system. Replaces and substantially extends qc-reflect.

Purpose: qc-align audits the code system against the corpus to identify and resolve redundancy, inconsistency, conceptual drift, corpus integrity issues, and coding gaps. It uses the LLM as an external auditor with full visibility of codebook documentation, corpus evidence, and tree structure. It is the primary tool for maintaining a lean, coherent, consistently-applied code system.

Position in the suite: - Upstream inputs: qc-scheme (documentation), qc-refactor (structural history), corpus files, SQLite database - Downstream outputs: qc-refactor queue (structural suggestions), SQLite batch writes (code propagation), corpus file edits (line splitting)

Audit modes:

Mode Object Primary signal Output
Overlap Code pairs Co-occurrence + doc similarity Merge/rename → refactor queue
Inconsistency Individual codes Excerpt vs. documentation mismatch Reassign/redefine suggestions
Bloat Sibling groups Redundant children under same parent Merge/delete → refactor queue
Restructure Branches or full tree Structural coherence Tree diff → refactor queue
Line splitting Corpus lines Topically composite lines File edits via qc corpus update
Code propagation Adjacent lines Missing codes in conversational context SQLite batch writes

Additional modes may be added in future iterations.

Prompt strategy: Every prompt includes a hierarchy context block — ancestry paths with scope + rationale at each level — to ground LLM assessment in the intended analytical framework. Parent and grandparent documentation is always included alongside the codes under review.

Candidate selection: User-driven. The scope picker is a multi-select tree where the user selects individual codes, sets, or branches.

Corpus format: All corpus lines have a Speaker: text format. Line splitting always preserves the speaker prefix on each resulting line. Lines are never merged across speakers. Code propagation may propose codes across speaker boundaries based on conversational context.

Backup and snapshot policy: Before any operation that modifies the database, corpus files, or codebook: SQLite backup to qc/backups/ and codebook snapshot via /snapshots/create. Both recorded in the session log.

Dryrun mechanism: All destructive operations show an exact preview before execution. No writes until explicit confirmation.

LLM model assignment: - Reasoning-intensive modes (Overlap, Inconsistency, Bloat, Restructure): qwen3:35b - Generative modes (Line splitting, Code propagation): qwen3:8b - Configurable per-mode in qc-atelier-config.yaml under align:

Refactor integration: Accepted structural suggestions pre-populate the qc-refactor queue via POST /refactor/queue. No direct execution from qc-align.

Code propagation and SQLite: The only place in qc-atelier where the database is written to directly rather than through the qc CLI. No qc codes apply command exists upstream. Mitigated by mandatory backup, dryrun, and session logging. To be revisited if a programmatic apply command is added upstream.

qc-unfold

Role: Active. LLM-assisted corpus meaning-making.

Purpose: qc-unfold derives interpretive outputs from the corpus using the cleaned, coherent code system as an analytical lens. It is the later analytical phase — after the code system has been calibrated by qc-align, qc-unfold uses it to generate meaning. Planned features include mini-biographies and relationship maps for named individuals and organisations, multi-axis summaries (by document, by code branch, by individual, by group), figuration detection and classification, and cross-speaker thematic association.

Status: Planned. Design pending completion of qc-align.

Shared infrastructure

qc-shared.lua

Loaded via dofile() at the top of every active filter. Key functions: read_yaml_file, meta_to_lua, S (stringify Pandoc metadata), N (numericise), read_text_file, get_json_files, read_json_file, to_json, parse_codebook_yaml, flatten_codebook, build_use_counts.

Shared path resolution pattern (applied to all active filters):

local function _get_project_root_early()
  local s = os.getenv("PANDOC_SCRIPT_FILE") or ""
  local r = s:gsub("/qc-atelier/[^/]+/[^/]+$", "")
  if r ~= "" and r ~= s then return r end
  local h = io.popen("pwd"); local cwd = h:read("*l"); h:close()
  return cwd or "."
end
local _shared_path = _get_project_root_early() .. "/qc-atelier/shared/qc-shared.lua"
local shared = dofile(_shared_path)

The dofile() approach was chosen over Lua’s require() because dofile takes an absolute path constructable from PANDOC_SCRIPT_FILE, avoiding Lua module path configuration in the Pandoc environment.

qc-shared.css

Shared CSS design tokens. Inlined before every tool’s own CSS. Two qc-viz-specific tokens (--viz-filter-warn-bg, --viz-filter-error-bg) live in qc-viz.css rather than the shared file. Autocomplete dropdown styles added to qc-shared.css.

qc-shared.js

Shared runtime JavaScript. Inlined before every tool’s own JS by the Lua filter. Currently contains qcAutocompleteInit(codeNames) — the @ code autocomplete system used by qc-scheme and qc-refactor.

@ autocomplete: Typing @ in any text input or textarea opens a floating dropdown of matching code names. Filters by prefix match first, then substring. Arrow keys navigate, Enter/Tab selects, Escape dismisses. The @query is replaced in-place with the selected code name. Reinitialised after every tree refresh so the code list stays current.

qc-atelier-server.py

Single Python HTTP server (stdlib only). Started from project root: python3 qc-atelier-server.py. Default port 8080. SERVE_DIR = qc/. SNAPSHOTS_DIR = qc/snapshots/.

qc-atelier-config.yaml

Unified config. QC_ATELIER_CONFIG env var overrides default path. Key top-level sections: directories, code_schema, server (includes qc_bin, port), ollama, candidates, viz, align.

qc/index.html

Static launcher served at /. Cards for scheme, viz, refactor (active), align (in development), reflect/unfold (planned stubs).

Early March 2025 — qc-viz initial build (chat 812fff87)

1.1 — Identifying the need for a corpus reader

Context. Working on qc-reflect, the _quarto.yml and directory listing of qc/json/ had just been shared, showing ten per-document coding files (interview dates and respondent initials as filenames). The question of how to pull data into an audit tool prompted a side decision about which JSON format to use.

Exchange. Asked whether to build a new format or reuse the existing qc/json/ exports. Those files were already produced by the pre-render script with qc codes find --json --pattern "$basename" --before 0 --after 0. Asked to see the shape of the JSON. Declined to share content directly (“I don’t want to paste actual content, it’s sensitive data”), so a Python one-liner was provided to print just the key structure. It was run and confirmed the per-document list-of-objects shape with {code, document, line, text} fields.

Decision: reuse qc/json/*.json as the data source for any corpus-facing tool. Rationale: the files already existed, were up to date, and contained the text field with full coded segment content. No new export format needed.

1.2 — Separating corpus visualisation from the audit tool

Decision: build qc-viz as a separate tool from qc-reflect, not as a panel inside it. The initial framing was qc-reflect as a unified audit interface, but line-by-line corpus reading (what qc-viz does) is a different concern from LLM-assisted code analysis (what qc-reflect does). Agreed to keep them separate, following the qc philosophy of single-purpose tools.

1.3 — Architecture: static HTML, no server

Decision: qc-viz would be purely static — all data baked into HTML at render time, no runtime server calls. This was a deliberate contrast to qc-scheme (which requires a server for read/write of codebook.docs.json). The rationale: qc-viz is read-only. It displays what has already been coded; it writes nothing back. A server dependency would add startup friction for what is essentially a document viewer. This made qc-viz the only tool in the ecosystem with no server component.

Implication accepted: any change to corpus files or coding requires a re-render. Confirmed acceptable given that re-render is already the workflow trigger for qc-viz.

1.4 — Pre-render script and data pipeline

Decision: use a bash pre-render script (qc-viz-pre-render.sh) to generate qc/json/*.json before the Lua filter runs. The script activates the Python venv, iterates over qc/corpus/*.txt, checks for files in qc/corpus/exclude/ to skip, and calls qc codes find --json for each.

Bug introduced. At some point during initial development the bash script was accidentally duplicated — a second #!/bin/bash shebang and a copy of the inner loop were appended. The second copy lacked the config-reading logic present in the first half. Not caught during initial development; remained until Step 3 of the unification work (2026-03-25).

1.5 — Config file: qc-viz-config.yaml

Decision: qc-viz gets its own config file, separate from qc-reflect-config.yaml. At the time, the ecosystem config was named qc-reflect-config.yaml and was understood as scoped to qc-reflect and qc-scheme. qc-viz was being developed independently. The config was given a qc-viz-config.yaml file with its own directories, files, code_filters, code_schema, display, codebook, export, ui, and advanced sections.

Known issue introduced. This created duplication: corpus_dir, json_dir, and exclude_dir were defined in both config files. Flagged as a known divergence; marked for resolution in the unification work.

1.6 — Code filtering system

Decision: implement a whitelist/blacklist filtering system in the Lua filter to control which codes appear in the visualisation. Motivation: deprecated or open-coding codes in the codebook should not appear in a clean visualisation. The blacklist was populated with a list of O_*-prefixed open-coding codes.

The filter logic operates on four matching mechanisms: exact code string, two-digit prefix, prefix range, and Lua regex pattern. Whitelist mode (if enabled) takes precedence over blacklist mode.

1.7 — Johnny Decimal prefix system for category display

Decision: use the first two digits of codes (the Johnny Decimal prefix) as the grouping unit for the filter panel. Each prefix gets a colour from a 20-colour rotating palette, and a label derived from: codebook (if integration enabled) → config categories map → built-in default schema → auto-generated from first code in prefix.

The codebook.enabled flag. Codebook integration was implemented but defaulted to false. Rationale: the codebook YAML parser was written but the integration was not yet validated against the actual codebook.yaml structure. Enabling by default risked silently wrong category labels.

1.8 — UI design

Decision: full-width filter panel at top, one collapsible section per interview below. Each section contains a four-column table (line number, speaker, text, codes) with all lines rendered — both coded and uncoded. Uncoded lines are hidden by default (toggle in filter panel). Gap rows show counts of hidden lines between visible ones.

Speaker column width. Computed dynamically: max speaker name length × 8 pixels, clamped to [80, 200]. Chosen to avoid column layout breaking on long participant identifiers without hardcoding a fixed width.

Stats bar. Each interview section has a summary row below the table showing visible/hidden counts, broken down by coded vs uncoded. Updated on every filter change via updateStats().

1.9 — Export system

Decision: implement CSV, JSON, Excel, HTML, and PDF exports client-side, with filter parameters documented in the output. A researcher filtering to specific codes and speakers should be able to export exactly what they see, with a record of what filter was applied. The filter parameter documentation was treated as a methodological transparency feature.

CDN dependency accepted. SheetJS and jsPDF are loaded from cdnjs.cloudflare.com unconditionally at page load. CSV, JSON, and HTML exports have no library dependency and work offline. Excel and PDF require the CDN. Accepted on the grounds that the research environment has internet access.

1.10 — localStorage persistence

Decision: persist all filter state to localStorage under the key "qc-viz-state". Rationale: the filter panel has enough state (per-code checkboxes, per-speaker checkboxes, uncoded toggle, section collapse states) that losing it on every reload would be frustrating. Save is synchronous (no debounce) because the state object is small.

Key not namespaced. The key "qc-viz-state" was chosen without a tool prefix. Flagged as a future migration task.

1.11 — Unimplemented config keys

Several config keys were defined in qc-viz-config.yaml during design but never wired up in the Lua filter or JS:

  • display.sections.default_collapsed — categories always start collapsed regardless
  • display.code_tags.show_full_code — full code always shown regardless
  • The entire export block — timestamp format, filename prefix
  • The entire ui block — includes persistence.storage_key, which was hardcoded as "qc-viz-state" in JS instead of being read from config

Decision: leave these as stubs. Rationale: the config keys were included during design to make the file self-documenting (showing what could be configured), but implementing them all before first use was over-engineering. Flagged in the config file with comments.

2026-04-24 — qc-align technical specification

Scope. Six audit modes: Overlap, Inconsistency, Bloat, Restructure, Line splitting, Code propagation. Additional modes deferred to future iterations.

Sparse and Undocumented modes excluded. Sparse codes are visible in qc-scheme via use counts in the sidebar — not an alignment problem. Undocumented codes are a qc-scheme workflow issue. Neither requires LLM analysis.

Hierarchy context in prompts. All prompts include parent and grandparent documentation (scope + rationale) as a framing block. Rationale: a flat list of code names gives the LLM insufficient context to reason about analytical boundaries. The parent’s documented scope defines what the category is supposed to carve out, which grounds assessment of whether children are doing their job and whether siblings are genuinely distinct.

Candidate selection is user-driven. Auto-detection heuristics (co-occurrence threshold, minimum sibling count) were considered and rejected as arbitrary. The user selects the scope via a multi-select tree picker, which drives candidate construction for any mode.

Line splitting and code propagation are separate modes. Both involve sequential reading of the corpus but touch different things: line splitting modifies corpus files (via qc corpus update); code propagation modifies the database (via SQLite batch write). Treating them as the same operation would blur the distinction between corpus integrity and coding completeness.

Line splitting is full-corpus, on-demand. Scanning only lines with N+ codes was considered and rejected as arbitrary. The LLM determines whether a line is topically composite; the user triggers the scan explicitly when ready.

Speaker prefix constraint. All corpus lines have a Speaker: text format. Line splitting always preserves the speaker prefix on each resulting line. Merging lines across speakers is never permitted. Code propagation may propose codes across speaker boundaries based on conversational context, but never merges transcript lines.

Backup and snapshot policy. Every destructive operation (database write, corpus file edit, structural change) creates both a timestamped SQLite backup and a codebook snapshot before execution. Rationale: qc-align can substantially reshape the code system across multiple audit sessions; restore points must exist at every intervention, not just at named milestones.

Dryrun mechanism. All destructive operations show an exact preview before execution. Rationale: the consequences of bulk code propagation or line splitting are significant and not easily reversible without the backup mechanism; users must see exactly what will change before confirming.

Model assignment. Reasoning-intensive modes (Overlap, Inconsistency, Bloat, Restructure) use qwen3:35b. Generative modes (Line splitting, Code propagation) use qwen3:8b. Rationale: the reasoning modes require careful analytical judgment about code boundaries and relationships; the generative modes are primarily reading and segmentation tasks where a smaller model performs adequately at significantly lower latency.

Refactor integration. Accepted structural suggestions (Overlap, Bloat, Restructure) pre-populate the qc-refactor queue via POST /refactor/queue. No direct execution from qc-align. Rationale: structural changes to the code system should be reviewed in qc-refactor before execution, maintaining the principle that qc-refactor is the single point of structural execution.

Code propagation and direct SQLite writes. This is the only place in qc-atelier where the database is written to directly rather than through the qc CLI. There is no qc codes apply CLI command. Mitigated by mandatory backup, dryrun preview, and session logging. This should be revisited if a programmatic apply command is added to qc upstream.
